# Архитектура проекта JSW Analysis Web

**Версия документа:** 1.0  
**Дата:** 29 мая 2026  
**Цель:** Обоснование выбранной структуры проекта для веб-версии системы анализа JSW (Joint Space Width).

---

## Почему именно такая структура?

Проект разрабатывается как **многопользовательское веб-приложение** на Streamlit.  
Ключевые требования:
- Поддержка одновременной работы многих пользователей
- Безопасная изоляция данных между пользователями и задачами
- Возможность фоновой обработки тяжёлых задач (инференс нейросети)
- Читаемость, расширяемость и поддерживаемость кода

---

## Полная структура проекта и обоснование каждого элемента

### Корневые файлы

| Файл                  | Назначение | Обоснование |
|-----------------------|----------|-----------|
| `main.py`             | Главный файл Streamlit | Точка входа приложения. Содержит только UI и высокоуровневую оркестрацию. Не должен содержать сложную логику. |
| `requirements.txt`    | Зависимости | Явный список всех библиотек |
| `.gitignore`          | Игнорирование файлов | Защита от коммита модели, временных файлов, кэша |
| `README.md`           | Документация проекта | Инструкции по запуску, настройке, развертыванию |
| `.env`                | Переменные окружения | Для хранения секретных параметров в будущем |
| `run.bat`             | Удобный запуск (Windows) | `streamlit run main.py` одной кнопкой |

---

### `core/` — Чистая нейросетевая логика

**Назначение:** Всё, что напрямую связано с машинным обучением.

- `__init__.py`
- `model_loader.py` — Загрузка и кэширование модели (`@st.cache_resource`)
- `preprocess.py` — YOLO-кроп, нормализация, подготовка изображения
- `inference.py` — Запуск U-Net, получение маски
- `visualization.py` — Расчёт метрик JSW + генерация визуализации

**Почему отдельно?**  
Эта папка должна быть максимально стабильной. Изменения здесь должны быть редкими и хорошо протестированными (медицинский софт).

---

### `services/` — Бизнес-логика и оркестрация

**Назначение:** Основная "умная" часть приложения.

- `session_service.py` — Создание, получение и очистка сессий пользователей
- `job_service.py` — Управление задачами (создание job, получение статуса, прогресс, отмена)
- `result_service.py` — Сохранение, получение и упаковка результатов анализа

**Почему важно вынести в `services/`?**  
Это предотвращает "спагетти-код" в `main.py`. Здесь сосредоточена вся бизнес-логика. Легко тестировать и расширять.

---

### `utils/` — Мелкие вспомогательные функции

- `constants.py` — Все константы проекта (размер изображения, device, пороги и т.д.)
- `helpers.py` — Чистые функции (например, `extract_metadata()`, валидация изображений)

**Правило:** Если функция ничего не знает про `st.session_state`, пути сессий или jobs — она идёт в `utils/`.

---

### `jobs/` — Система фоновой обработки

- `job_queue.py` — In-memory очередь задач + управление worker'ами

**Почему нужен отдельный слой?**  
Инференс нейросети — тяжёлая операция. Запускать её напрямую в Streamlit-скрипте опасно (блокировка интерфейса, проблемы с многопоточностью). Этот слой позволит:
- Запускать обработку в фоне
- Показывать прогресс-бар в реальном времени
- Поддерживать отмену задания
- Масштабироваться в будущем (переход на Celery/Redis)

---

### `temp/sessions/{session_id}/{job_id}/` — Временное хранилище

**Самая важная часть архитектуры для multi-user.**

Каждый пользователь → своя сессия → каждое задание → свой `job_id`.

Пример пути:
`temp/sessions/a1b2c3d4-e5f6-.../job_98765/`

Внутри job_id хранятся:
- исходные изображения
- препроцессированные файлы
- маски
- результаты визуализации
- CSV-отчёт

**Это гарантирует полную изоляцию** данных между пользователями и между разными запусками одного пользователя.

### Другие папки

| Папка           | Назначение |
|-----------------|----------|
| `models/`       | Хранение весов нейросети (`best_model.pth`) |

## Принципы архитектуры

1. **Separation of Concerns** — чёткое разделение ответственности
2. **Isolation** — каждый пользователь и каждое задание изолированы
3. **Scalability** — легко перейти от in-memory очереди к Celery + Redis
4. **Testability** — core и services можно покрыть тестами
5. **Maintainability** — понятная структура даже через 6–12 месяцев
